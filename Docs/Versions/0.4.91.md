# LOS 0.4.91

This update removes the misleading early scheduler stack-pool failure while keeping the current stable timer path intact.

## Summary
- bumped the archive version to 0.4.91
- stopped the early scheduler stack-pool reservation from probing the memory manager while that hosted bootstrap path is intentionally disabled
- kept the scheduler on the direct-claim stack-pool path until memory-manager-backed thread stacks are explicitly re-enabled later
- replaced the visible bootstrap failure with a plain informational trace on the deferred path

## Why this change
The timer proof is now stable, so the remaining visible `[FAIL]` was no longer useful. It came from an early stack-pool reservation attempt that should not have been happening while `SchedulerMayUseMemoryManagerBackedThreadStacks()` still returns false. This update makes the reserve path follow that same policy, so the kernel goes straight to the working direct-claim path instead of reporting a bootstrap failure first.
